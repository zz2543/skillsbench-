01 · 背景与问题

为什么要研究"库化技能"

技能（Agent Skill）是注入给 LLM 智能体的可复用过程知识包。SkillsBench 证明：人工技能平均把任务通过率 提高约 16 个百分点；但它的原生协议是"每个任务只装它自己的技能"——不存在真实部署里的库化检索环节。 真实部署中，用户会装几十上百个来源各异的技能，系统靠约 30 token 的元数据检索、注入多个技能， 多技能共存时可能相互干扰（指令矛盾、功能冗余、上下文稀释、顺序敏感）。

现象

库化检索会损失多少性能？负增益任务是否增多？

本阶段目标

先做最朴素的一版：全量库化 + agent 自选，不做任何冲突处理，量出基线变化。

对照基准

官方 DeepSeek V4 Pro 榜单分（任务自带技能 = oracle 协议）。

两个条件（同一 10 任务）

  • no-skill：不注入任何技能（floor）——--skill-mode no-skill
  • library：注入全量 195 技能库，agent 自选——--skill-mode with-skill --skills-dir data/library

技能是怎么"自选"的

OpenHands 把库里每个技能以名称+简介暴露为 invoke_skill(name=…) 工具，模型按语义自行调用—— 这正是 Anthropic 渐进式披露、也是真实库化部署的形态。库化后 195 个候选同时可见。

03 · 工程搭建

把官方基准在本机（中国网络）跑通

最大的工程量不在"调用 API"，而在让 Docker 沙箱在 Clash（fake-ip/TUN）代理下完成 镜像构建 + OpenHands 自装 + verifier 的全链路网络。三处配合后 oracle 冒烟通过（reward=1）。

诊断到的三类网络问题

  • ubuntu/debian 官方 apt 镜像被 fake-ip 路由，返回 502（80/87 任务镜像受影响）
  • astral.sh 的 uv 安装脚本 SSL_ERROR_SYSCALL
  • github 直连失败，但经 Clash HTTP 代理可用；aliyun 直连可用

解决方案（三处）

  • 代理桥：0.0.0.0:7898 → 127.0.0.1:7897 暴露 loopback-only 的 Clash
  • Docker 代理：HTTPS 走代理，noProxy 让 aliyun 直连
  • 基础镜像补丁：apt→aliyun HTTP + 预装 uv，OpenHands 自装遂稳定成功

凭证接线（DeepSeek → OpenHands via benchflow）

--model deepseek/deepseek-v4-pro  ·  --agent-env DEEPSEEK_API_KEY=…  ·  DEEPSEEK_BASE_URL=api.deepseek.com/v1

benchflow 把 DEEPSEEK_API_KEY → BENCHFLOW_PROVIDER_API_KEY → 映射为 OpenHands 的 LLM_API_KEY/LLM_BASE_URL； 带 deepseek/ 前缀才能让 host 侧用量计费路由到 deepseek provider 而非 openai。密钥全程 gitignore、日志打码。

07 · 局限与下一步

本阶段结论与后续

局限

  • 10 任务、每条件 1 次重复——量级信号，非统计定论
  • 库化=全量 195 技能同时可见，未做任何检索/裁剪
  • 与官方为同 harness、同模型，但技能供给协议不同（自选 vs 自带）

下一步（CASL）

  • 两两交互测量 → 稀疏干扰矩阵
  • 组合感知加载：静态筛查 → 定向验证 → 冲突图子集选择
  • 目标：消除库化损失、修复负增益任务
